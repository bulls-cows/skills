# 文件修改范围限制规则

## 核心原则

| 行为 | 说明 |
| ---- | ---- |
| ✅ 允许 | 修改用户授权目录范围内的文件 |
| ✅ 允许 | 在授权目录内创建新文件 |
| 🚫 禁止 | 修改授权目录范围外的任何文件 |
| 🚫 禁止 | 在授权目录范围外创建新文件 |
| 🚫 禁止 | 删除任何授权目录范围外的文件 |

---

## 授权机制

### 默认行为

AI 助手可以修改的**默认目录范围**由以下方式确定：

1. **首次对话时**：AI 应主动询问用户希望限制在哪些目录
2. **用户未指定时**：AI 仅允许修改用户当前对话中**明确涉及**的文件或目录
3. **跨目录操作**：当需要操作默认范围外的文件时，必须获得用户**明确授权**

### 授权类型

| 授权类型 | 说明 | 示例 |
| -------- | ---- | ---- |
| 目录授权 | 授权修改指定目录下的所有文件 | `/src/components`、`/src/api` |
| 文件授权 | 授权修改指定的单个文件 | `/src/utils/helper.ts` |
| 通配符授权 | 使用通配符授权某类文件 | `/src/**/*.ts` |

### 授权获取方式

当 AI 需要修改非授权目录的文件时，必须：

1. **停止操作** - 不要直接修改文件
2. **明确告知** - 向用户说明：
   - 需要修改的文件路径
   - 修改的目的/原因
   - 需要的授权类型（目录/文件/通配符）
3. **等待确认** - 只有获得用户明确同意后才能执行

---

## 模式说明

### Plan Mode（计划模式）

1. AI 分析需求，列出需要修改的所有文件
2. 检查每个文件是否在授权范围内
3. 对于非授权范围内的文件，在计划中标记为"需要授权"
4. 将完整计划展示给用户，等待确认后再执行

### Spec Mode（规格模式）

同计划模式，所有需要修改的文件都必须在规格说明中明确列出。

### Auto Mode（自动模式）

1. 仍然遵循授权限制
2. 遇到非授权文件时，暂停并请求授权
3. 不能跳过任何未授权的文件

## 授权示例

### 场景 1：用户在对话开始时说

> "帮我修改 /src/components 目录下的 Button 组件"

授权范围：`/src/components`

✅ AI 可以直接修改 `/src/components/Button.tsx`
❌ AI 不能修改 `/src/utils/format.ts`（未授权）

### 场景 2：AI 需要修改非授权文件

AI 试图修改 `/src/api/user.ts`：

```text
⚠️ 警告：/src/api/user.ts 不在授权范围内
- 当前授权目录：/src/components
- 需要修改的原因：[说明原因]
- 是否授权修改此文件？
```

### 场景 3：用户要求跨目录操作

用户说："顺便把 utils 目录下的日期格式化函数也改一下"

这视为**新的授权请求**，AI 需要确认：

```text
📋 新的授权请求：
- 目录：/src/utils
- 原因：修改日期格式化函数
- 请确认是否授权
```

## 授权管理

### 动态授权

- 用户可以随时扩大或缩小授权范围
- 扩大授权需要用户明确确认
- 缩小授权（撤销）立即生效

### 授权记录

AI 应该在对话中保持清晰的授权记录，例如：

```text
📂 当前授权目录：
- /src/components
- /src/utils

📄 当前授权文件：
- /src/config/app.ts
```

---

## 注意事项

1. **不信任默认假设**：除非用户明确说明，否则不要假设用户授权了任何目录
2. **最小权限原则**：默认只授权当前对话中明确涉及的文件/目录
3. **安全敏感文件**：即使在授权范围内，涉及安全（如认证、密钥、配置文件）的操作仍需额外谨慎
4. **删除操作**：删除文件是高度敏感操作，即使在授权范围内，也建议先确认
