# Lint and Commit 技能

此技能组合了代码质量检查和 Git 提交流程，确保每次提交的代码都经过质量检查。

## 工作流程

### 第一阶段：代码质量检查

#### 步骤 1：检查 Node.js 版本

1. 读取项目根目录下的 `.nvmrc` 文件，获取要求的 Node.js 版本
2. 执行 `node --version` 命令获取当前 Node.js 版本
3. 比较当前版本是否不小于 `.nvmrc` 中记录的版本

**如果版本不满足要求：**

- 显示当前版本和要求版本信息
- 提示用户升级 Node.js 到要求的版本或更高版本
- **停止整个流程，不执行 lint 和 commit**

**如果版本满足要求：**

- 继续执行步骤 2

#### 步骤 2：执行 lint 检查

执行 `npm run lint` 命令检测代码风格和 TypeScript 类型。

**情况 1：lint 通过，无错误**

- 显示成功消息
- 继续进入第二阶段（Git 提交流程）

**情况 2：lint 发现错误**

- 显示错误信息
- 尝试自动修复错误（遵循 lint 技能的修复原则）
- 修复后再次运行 `npm run lint` 验证
- 如果修复成功，继续进入第二阶段
- 如果修复失败，**停止流程**，报告无法修复的错误

#### Lint 修复原则

修复 lint 报错时必须**正面解决问题**，不得回避：

- **不能**随意给变量添加 `any` 类型声明来绕过类型检查
- **不能**删除未开发完毕的功能代码来消除报错
- **应当**理解报错的实际含义，针对根本问题进行修复

### 第二阶段：Git 提交流程

Lint 检查通过后，执行标准的 commit 流程。

#### 步骤 1：分析当前状态

并行运行以下命令了解当前状态：

```bash
git status
git diff
git diff --staged
git log --oneline -5
```

#### 步骤 2：理解改动意图

综合以下信息理解此次提交的目的：

1. **代码变更分析**：从 diff 中识别新增功能、修复问题、优化重构或文档更新
2. **对话上下文**：提取用户在对话中提到的功能需求或改动意图
3. **变更范围**：确定影响的模块或功能区域

#### 步骤 3：智能选择暂存文件

根据以下原则选择需要暂存的文件：

**应该暂存的文件：**

- 源代码文件（.ts, .js, .vue, .jsx, .tsx 等）
- 配置文件（package.json, tsconfig.json 等）
- 文档文件（.md, .txt 等）
- 样式文件（.css, .scss 等）
- 测试文件

**应该警告的文件（询问用户）：**

- 环境变量文件（.env, .env.local 等）
- 凭证文件（credentials.json, secrets.\* 等）
- 私钥文件（_.key, _.pem 等）
- 大型二进制文件
- node_modules 或其他依赖目录

**应该忽略的文件：**

- 构建产物（dist/, build/ 等，除非明确需要）
- 临时文件（_.tmp, _.swp 等）

如果发现需要警告的文件，明确告知用户并询问是否应该包含。

#### 步骤 4：生成提交信息

提交信息必须遵循传统格式：`type(scope): description`

**Type（类型）：**

- `feat` - 新功能
- `fix` - 修复 bug
- `docs` - 文档更新
- `style` - 代码格式调整（不影响功能）
- `refactor` - 重构（既非新功能也非修复）
- `perf` - 性能优化
- `test` - 测试相关
- `chore` - 构建/工具/依赖相关
- `revert` - 回滚提交

**Scope（范围）- 可选：**

- 受影响的模块、组件或功能区域
- 例如：`auth`、`editor`、`api`、`ui`

**Description（描述）：**

- 使用中文（代码标识符、专有名词除外）
- 使用动词开头的祈使语气
- 精炼，不超过 50 个字符
- 准确描述"做了什么"而非"怎么做的"

**特别提示**：如果此次提交包含了 lint 自动修复的改动，应该在提交信息中体现。例如：

- `style: 修复代码格式问题` - 如果主要是格式修复
- `fix(auth): 修复类型错误并完善认证逻辑` - 如果包含类型修复和功能改动
- 或者在原有提交类型基础上，准确反映实际改动

#### 步骤 5：展示并确认

在执行提交前，向用户展示：

1. **Lint 检查结果摘要**（通过或修复了哪些问题）
2. **将要暂存的文件列表**
3. **关键变更摘要**
4. **生成的提交信息**

使用清晰的格式展示，例如：

```
✅ 代码质量检查通过

即将提交以下更改：

📝 暂存文件：
  - src/auth/login.ts
  - src/components/LoginForm.vue
  - docs/api.md

📊 主要变更：
  - 新增 JWT token 生成逻辑
  - 添加登录表单验证
  - 更新 API 文档

💬 提交信息：
feat(auth): 添加 JWT 用户认证功能

是否确认提交？
```

#### 步骤 6：执行提交

用户确认后，按顺序执行：

**重要提示**：在提交时，你必须动态填写当前使用的 AI 模型名称，而不是硬编码。根据你的系统提示中的模型信息，填写真实的模型名称。

```bash
# 1. 暂存选定的文件
git add <file1> <file2> ...

# 2. 创建提交（使用 HEREDOC 确保格式正确）
# 注意：将 <当前 AI 模型名称> 替换为你实际使用的模型名称
git commit -m "$(cat <<'EOF'
<提交信息>

Commited using model: <当前 AI 模型名称>
EOF
)"

# 3. 确认提交成功
git status
git log -1
```

## 特殊情况处理

### Lint 检查失败且无法自动修复

如果 lint 检查失败且自动修复无效：

```
❌ Lint 检查发现以下无法自动修复的错误：

[显示错误信息]

请手动修复这些错误后再重新运行此技能，或者直接运行 /commit 跳过 lint 检查。
```

### 多个独立改动

如果发现多个不相关的改动，建议用户分别提交：

```
我注意到此次变更包含多个独立的改动：
1. 新增用户认证功能
2. 修复编辑器保存 bug

建议分别提交以保持提交历史清晰。你希望如何处理？
```

### 无变更内容

如果 git status 显示没有变更：

```
当前工作区没有未提交的变更。

如果你期望有变更，可能的原因：
- 文件尚未保存
- 变更已经在之前的提交中
- .gitignore 忽略了这些文件
```

### 并行任务正在执行

如果当前有多个任务正在并行执行（多个子任务同时修改文件）：

```
⚠️ 检测到有并行任务正在执行，建议等所有任务完成后再执行 lint 和提交，
以避免文件冲突。

是否继续？
```

## 注意事项

1. **完整的质量保证**：只有在 lint 完全通过后才会执行提交
2. **保持交互性**：在关键步骤（文件选择、提交确认）保持用户参与
3. **清晰的状态反馈**：明确告知每个阶段的执行状态
4. **遵守 Git 规范**：从不跳过 hook，不使用强制操作
5. **提交信息质量**：确保提交信息准确反映改动内容（包括 lint 修复）

## 与独立技能的关系

- 此技能内部执行 lint 和 commit 的完整流程
- 用户仍然可以单独使用 `/lint` 或 `/commit` 技能
- 当用户明确表示想要"先检查再提交"时，使用此组合技能更高效
