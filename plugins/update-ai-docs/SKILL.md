# Update AI Documentation

这个技能用于总结编码过程中的最佳实践经验并更新到项目文档中，供 AI 参考，避免再犯类似错误。

## 使用方式

用户可以通过以下方式调用这个技能：

- `/update-ai-docs` - 引导用户输入要总结的内容
- `/update-ai-docs <经验总结内容>` - 直接处理用户提供的内容

## 工作原则

文档体系采用两层结构：

- **`docs/` 目录**：存放具体、详细的内容（每个文件聚焦一个主题）
- **`AGENTS.md`**：总纲，通过引用链接指向 `docs/` 文件，不写重复内容

**更新时优先找到最合适的现有文档来添加内容，而不是创建新文件；只有当内容主题在现有文件中找不到归属时，才新建文档。**

## 工作流程

### 步骤 1：获取要总结的内容

如果用户没有提供具体内容，询问用户要总结的最佳实践经验：

```
请描述你想要总结到项目文档中的最佳实践经验或规范：
1. 具体的问题或错误
2. 解决方案或最佳实践
3. 相关的代码示例（可选）
4. 建议归类到哪个章节（可选）
```

并明确：

- 内容类型：最佳实践 / 规范约定 / bug修复 / 架构决策 / 其他
- 适用范围：全局 / 特定功能模块 / 特定技术栈
- 核心要点：用一句话总结这条知识

### 步骤 2：检查 AGENTS.md 文件

检查项目根目录下是否存在 `AGENTS.md` 文件：

**情况 1：AGENTS.md 不存在**

- 提示用户：

  ```
  项目根目录下没有找到 AGENTS.md 文件。

  建议创建该文件作为 AI 开发助手的主要参考文档。

  是否要立即创建 AGENTS.md 文件？[Y/n]
  ```

- 如果用户同意创建，创建一个基础的 AGENTS.md 模板，然后继续执行步骤 3
- 如果用户拒绝，终止任务并提示用户稍后手动创建

**情况 2：AGENTS.md 存在**

- 继续执行步骤 3

### 步骤 3：决定放置位置

读取 `AGENTS.md` 文件，分析其结构并列出 `docs/` 目录下所有文件。对每个现有文档，判断新内容是否属于其主题范围。

**情况A：找到合适的现有文档**

- 读取目标文档内容
- 找到合适的章节插入内容
- 如果没有合适的章节，创建新章节
- 更新文档
- 不需要修改 AGENTS.md（引用已存在）

**情况B：没有合适的现有文档**

- 在 `docs/` 目录下新建一个主题明确的文件（中文命名，如 `vue组件规范.md`、`api调用最佳实践.md`）
- 将内容写入新文件
- 在 AGENTS.md 中新增对该文件的引用

**情况C：内容适合直接补充到 AGENTS.md 的概要说明中**（仅适用于非常核心的架构级别内容，且内容很简短）

- 直接在 AGENTS.md 的相关章节添加简短说明
- 如果说明需要展开，还是走情况B的路径

### 步骤 4：格式化并插入内容

根据文档的现有格式，组织要插入的内容：

**内容结构：**

```markdown
### [章节标题]

**[子标题或要点]**

[详细描述]

**[示例/代码]**（如果适用）

\`\`\`[语言]
[代码示例]
\`\`\`

**[注意事项]**（如果适用）

- 注意事项1
- 注意事项2
```

**语言规范：**

- 文档主体使用中文
- 专有名词（`Vue`、`TypeScript`、`React`）、代码片段、已广泛使用的技术缩写（`API`、`SDK`、`OSS`）保留英文
- 正文叙述不夹杂不必要的英文，如"这个 component 需要 emit 一个 event"应改为"该组件需要通过事件通知父级"

**插入位置优先级：**

1. 如果存在明确相关的章节，插入到该章节内
2. 如果是新的主题，创建新的章节
3. 保持文档的层级结构一致性

### 步骤 5：更新文档并确认

1. 使用 Edit 或 Write 工具更新目标文档
2. 显示更新摘要：

   ```
   ✓ 已更新文档：[文件路径]
   ✓ 更新章节：[章节名称]
   ✓ 新增内容：[简要描述]

   内容已添加到 [章节路径]，供后续 AI 开发参考。
   ```

## 示例场景

**场景1：记录 Vue 组件规范**
用户说："记录一下：Vue组件中不要直接修改 props，要通过 emit 事件"
→ 查找是否有 Vue 组件相关文档 → 找到后追加这条规范 → 不改 AGENTS.md

**场景2：记录一个 bug 修复经验**
用户说："把这个坑记下来：iOS WebView 中 position:fixed 在软键盘弹起时会失效"
→ 查找是否有兼容性/移动端相关文档 → 没有则新建 `docs/移动端兼容性问题.md` → 在 AGENTS.md 中添加引用

**场景3：记录 API 调用约定**
用户说："所有 API 错误都要用统一的错误处理器，不能自己 catch 后静默忽略"
→ 查找 API 相关文档 → 找到合适文件后追加

## 最佳实践

1. **保持文档结构清晰**

   - 使用标准的 Markdown 层级结构
   - 章节标题要具有描述性
   - 使用列表和代码块增强可读性

2. **内容组织原则**

   - 将相似的规范/实践归类到同一章节
   - 提供具体的代码示例
   - 说明"为什么"而不仅仅是"怎么做"
   - 包含反例说明（如果适用）
   - **面向 AI 读者**：写的是给下一次对话中的 AI 看的，要清晰、无歧义、可操作
   - **具体而非抽象**：给出代码示例、文件路径、具体场景，而非泛泛而谈

3. **保持一致性**

   - 遵循现有文档的格式风格
   - 使用一致的术语和命名
   - 保持章节编号/层级的一致性

4. **避免重复**
   - 更新前先检查是否已存在类似内容
   - 如果已存在，考虑补充或改进现有内容，而不是添加重复内容

## 注意事项

- 在更新文档前，务必先读取文档内容，了解现有结构
- 不要删除或修改现有内容，除非用户明确要求
- 如果不确定应该放在哪个章节，询问用户的意见
- 保持专业和简洁的技术写作风格
- 使用中文编写内容（除非项目文档是英文）
- **不要将同一条内容写进多个文件（避免文档碎片化）**
- docs/ 文件的内容深度 > AGENTS.md 的引用说明深度
- 如果用户提供的内容已经在文档中存在，告知用户并说明在哪里，不重复写入
- 对于非常临时或版本相关的信息（如"某个依赖的临时 bug"），写入时标注"临时"或版本号
