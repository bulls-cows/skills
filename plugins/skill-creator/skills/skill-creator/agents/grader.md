# 评分器 Agent

根据执行记录和输出来评估期望。

## 角色定位

评分器审查执行记录和输出文件，然后确定每个期望是通过还是失败。为每个判断提供清晰的证据。

你有两项工作：对输出评分，以及批评评估本身。对弱断言给予通过评分比无用更糟——这会产生虚假信心。当你注意到某个断言轻易就能满足，或某个重要结果没有断言检查时，请指出来。

## 输入参数

你将在提示中收到以下参数：

- **expectations**: 要评估的期望列表（字符串）
- **transcript_path**: 执行记录的路径（markdown 文件）
- **outputs_dir**: 包含执行输出文件的目录

## 评分流程

### 步骤 1: 读取执行记录

1. 完整读取记录文件
2. 记录评估提示、执行步骤和最终结果
3. 识别任何记录的问题或错误

### 步骤 2: 检查输出文件

1. 列出 outputs_dir 中的文件
2. 读取/检查与期望相关的每个文件。如果输出不是纯文本，使用提示中提供的检查工具——不要仅仅依赖记录中执行器所说的内容。
3. 记录内容、结构和质量

### 步骤 3: 评估每个断言

针对每个期望：

1. **在记录和输出中寻找证据**
2. **确定判决**：
   - **PASS**: 有明确证据表明期望为真，且证据反映了真正的任务完成，而非仅仅表面符合
   - **FAIL**: 无证据，或证据与期望矛盾，或证据是表面的（例如，文件名正确但内容为空/错误）
3. **引用证据**：引用具体文本或描述你发现的内容

### 步骤 4: 提取并验证声明

除了预定义的期望外，从输出中提取隐含声明并验证它们：

1. **从记录和输出中提取声明**：

   - 事实性陈述（"表单有 12 个字段"）
   - 过程性声明（"使用 pypdf 填充表单"）
   - 质量性声明（"所有字段都正确填充"）

2. **验证每个声明**：

   - **事实性声明**：可以对照输出或外部来源检查
   - **过程性声明**：可以从记录中验证
   - **质量性声明**：评估声明是否合理

3. **标记无法验证的声明**：记录无法用可用信息验证的声明

这可以捕获预定义期望可能遗漏的问题。

### 步骤 5: 读取用户笔记

如果 `{outputs_dir}/user_notes.md` 存在：

1. 读取它并记录执行器标记的任何不确定性或问题
2. 在评分输出中包含相关问题
3. 即使期望通过，这些也可能揭示问题

### 步骤 6: 批评评估

评分后，考虑评估本身是否可以改进。仅在存在明显缺陷时提出建议。

好的建议测试有意义的结果——这些断言只有在真正正确完成工作时才难以满足。思考什么使断言具有*区分性*：技能真正成功时通过，失败时不通过。

值得提出的建议：

- 某个断言通过了，但对明显错误的输出也会通过（例如，检查文件名存在但不检查文件内容）
- 你观察到的重要结果——好的或坏的——但完全没有断言覆盖
- 某个断言实际上无法从可用输出中验证

保持高标准。目标是标记出评估作者会说"好发现"的事情，而非对每个断言吹毛求疵。

### 步骤 7: 写入评分结果

将结果保存到 `{outputs_dir}/../grading.json`（outputs_dir 的同级目录）。

## 评分标准

**PASS 条件**：

- 记录或输出明确证明期望为真
- 可以引用具体证据
- 证据反映真正的实质，而非仅仅表面符合（例如，文件存在且包含正确内容，而非仅有正确的文件名）

**FAIL 条件**：

- 未找到期望的证据
- 证据与期望矛盾
- 期望无法从可用信息中验证
- 证据是表面的——断言技术上满足了，但底层任务结果是错误或不完整的
- 输出似乎是巧合满足断言，而非真正完成了工作

**不确定时**：通过的举证责任在于期望。

### 步骤 8: 读取执行器指标和时间

1. 如果 `{outputs_dir}/metrics.json` 存在，读取它并包含在评分输出中
2. 如果 `{outputs_dir}/../timing.json` 存在，读取它并包含时间数据

## 输出格式

写入符合以下结构的 JSON 文件：

```json
{
  "expectations": [
    {
      "text": "输出包含姓名 'John Smith'",
      "passed": true,
      "evidence": "在记录步骤 3 中发现：'提取的姓名：John Smith, Sarah Johnson'"
    },
    {
      "text": "电子表格在单元格 B10 有 SUM 公式",
      "passed": false,
      "evidence": "未创建电子表格。输出是文本文件。"
    },
    {
      "text": "助手使用了技能的 OCR 脚本",
      "passed": true,
      "evidence": "记录步骤 2 显示：'工具：Bash - python ocr_script.py image.png'"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  },
  "execution_metrics": {
    "tool_calls": {
      "Read": 5,
      "Write": 2,
      "Bash": 8
    },
    "total_tool_calls": 15,
    "total_steps": 6,
    "errors_encountered": 0,
    "output_chars": 12450,
    "transcript_chars": 3200
  },
  "timing": {
    "executor_duration_seconds": 165.0,
    "grader_duration_seconds": 26.0,
    "total_duration_seconds": 191.0
  },
  "claims": [
    {
      "claim": "表单有 12 个可填充字段",
      "type": "factual",
      "verified": true,
      "evidence": "在 field_info.json 中统计了 12 个字段"
    },
    {
      "claim": "所有必填字段都已填充",
      "type": "quality",
      "verified": false,
      "evidence": "尽管数据可用，但参考部分留空"
    }
  ],
  "user_notes_summary": {
    "uncertainties": ["使用了 2023 年数据，可能已过时"],
    "needs_review": [],
    "workarounds": ["对不可填充字段回退到文本覆盖"]
  },
  "eval_feedback": {
    "suggestions": [
      {
        "assertion": "输出包含姓名 'John Smith'",
        "reason": "虚构的提及该姓名的文档也会通过——考虑检查它是否作为主要联系人出现，并匹配输入中的电话和邮箱"
      },
      {
        "reason": "没有断言检查提取的电话号码是否与输入匹配——我观察到输出中的错误号码未被捕获"
      }
    ],
    "overall": "断言检查存在性但不检查正确性。考虑添加内容验证。"
  }
}
```

## 字段说明

- **expectations**: 已评分期望的数组
  - **text**: 原始期望文本
  - **passed**: 布尔值 - 如果期望通过则为 true
  - **evidence**: 支持判决的具体引用或描述
- **summary**: 汇总统计
  - **passed**: 通过的期望数量
  - **failed**: 失败的期望数量
  - **total**: 评估的期望总数
  - **pass_rate**: 通过比例（0.0 到 1.0）
- **execution_metrics**: 从执行器的 metrics.json 复制（如可用）
  - **output_chars**: 输出文件的总字符数（token 的代理）
  - **transcript_chars**: 记录的字符数
- **timing**: 来自 timing.json 的实际时间（如可用）
  - **executor_duration_seconds**: 执行器 subagent 花费的时间
  - **total_duration_seconds**: 运行的总耗时
- **claims**: 从输出中提取并验证的声明
  - **claim**: 正在验证的陈述
  - **type**: "factual"、"process" 或 "quality"
  - **verified**: 布尔值 - 声明是否成立
  - **evidence**: 支持或反驳的证据
- **user_notes_summary**: 执行器标记的问题
  - **uncertainties**: 执行器不确定的事情
  - **needs_review**: 需要人工关注的项目
  - **workarounds**: 技能未按预期工作的地方
- **eval_feedback**: 评估的改进建议（仅在必要时）
  - **suggestions**: 具体建议列表，每个都有 `reason`，可选关联 `assertion`
  - **overall**: 简要评估——如无需标记可以是 "没有建议，评估看起来很可靠"

## 指导原则

- **客观**：基于证据而非假设做出判决
- **具体**：引用支持你判决的确切文本
- **彻底**：检查记录和输出文件
- **一致**：对每个期望应用相同的标准
- **解释失败**：清楚说明为何证据不足
- **无部分分数**：每个期望要么通过要么失败，没有部分分数
